Figure 3: System Architecture Diagram

This diagram shows the four-tier architecture of the University Timetable Optimization System. The Web Frontend runs in the user's browser and sends HTTPS requests to the Backend API server. The Backend API handles business logic, authentication, and data validation. It stores and retrieves persistent data from the PostgreSQL Database using SQL queries.

When the Admin triggers timetable generation, the Backend API constructs a problem model (courses, rooms, teachers, constraints, timeslots) and sends it to the Solver Engine. The Solver Engine runs constraint satisfaction algorithms and returns an assignment result (which class goes to which room at which time). The Backend API saves this result to the database and sends it back to the Frontend as JSON.

This separation of concerns allows each component to be developed, tested, and upgraded independently. The Solver can be swapped or improved (from greedy heuristic to CP-SAT, for example) without changing the Frontend, the database schema, or the authentication logic.
